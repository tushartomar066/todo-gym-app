## 1. What is this app?

**Jtrac** is a full-stack fitness + productivity tracker. It lets users:

- **Todo** — Create and manage daily tasks with priorities (low / medium / high)
- **Gym** — Log workouts: add exercises, track sets (weight + reps), view personal records
- **Cardio** — Log cardio sessions (run, walk, cycle, warmup, other) with distance, steps, duration
- **Dashboard** — See today's stats, weekly task completion chart, workout status

It is a **Progressive Web App (PWA)** — users can install it on mobile/desktop like a native app.

## 4. Environment Setup

### Prerequisites

- Node.js 18+
- A [Supabase](https://supabase.com) project (free tier works)

### Step 1 — Clone and install

```bash
git clone https://github.com/tushartomar066/todo-gym-app.git
cd todo-gym-app
npm install
```

### Step 2 — Create `.env.local`

Create a file called `.env.local` in the project root:

```env
NEXT_PUBLIC_SUPABASE_URL=https://your-project-id.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=your-anon-key-here
```

Find these values in your Supabase dashboard under **Project Settings → API**.

> These are `NEXT_PUBLIC_` prefixed because they're safe to expose to the browser (they use RLS for security, not secret keys).

### Step 3 — Set up the database

Run the SQL migration files in order against your Supabase project. Go to **Supabase Dashboard → SQL Editor** and paste each file:

1. `supabase/migrations/00001_initial_schema_and_rls.sql` — Creates profiles, tasks, workouts, exercises, sets tables with RLS
2. `supabase/migrations/00002_fix_schema.sql` — Schema fixes
3. `supabase/migrations/00003_fresh_setup.sql` — Clean setup
4. `supabase/migrations/00004_reset_and_setup.sql` — Reset and re-setup
5. `supabase/migrations/00005_cardio_log.sql` — Adds cardio_logs table
6. `supabase/migrations/00006_cardio_update_policy.sql` — RLS policy for cardio updates

### Step 4 — (Optional) Enable Google OAuth

In Supabase Dashboard → **Authentication → Providers → Google**, add your Google OAuth credentials. The callback URL is:

```
https://your-project.supabase.co/auth/v1/callback
```

## 6. Authentication

### How it works

Authentication is handled entirely by Supabase Auth with two methods:

1. **Email / Password** — standard sign in / sign up
2. **Google OAuth** — redirects through Google, comes back to `/auth/callback`

### Middleware (`proxy.ts`)

This file runs on **every request** as Next.js middleware. It:

1. Creates a Supabase server client using request cookies
2. Calls `supabase.auth.getUser()` to check if the user is logged in
3. If the route is protected (`/dashboard`, `/todo`, `/gym`) and user is NOT logged in → redirects to `/`
4. Updates auth cookies with a 3-day max age on each request (keeps session alive)

```
Request → proxy.ts → check auth → protected? → redirect to / OR continue
```

### Auth callback (`/auth/callback/route.ts`)

After Google OAuth, Google redirects to this route with a `code` parameter. The route:
1. Exchanges the code for a Supabase session
2. Sets the session cookie
3. Redirects the user to `/dashboard`

### Session cookies

The server-side Supabase client (`src/lib/supabase/server.ts`) reads/writes auth session data to HTTP cookies via Next.js `cookies()` API. Cookie max age is set to 3 days.

### Row-Level Security (RLS)

**Every table has RLS enabled.** Each table has policies like:

```sql
-- Users can only SELECT their own rows
CREATE POLICY "Users can view own tasks"
  ON tasks FOR SELECT
  USING (auth.uid() = user_id);

-- Users can only INSERT with their own user_id
CREATE POLICY "Users can insert own tasks"
  ON tasks FOR INSERT
  WITH CHECK (auth.uid() = user_id);
```

This means even if someone gets the anon key, they cannot read or write another user's data.

## 8. Server Actions (API Layer)

All database operations live in **`src/lib/actions.ts`** as Next.js Server Actions (`'use server'`). There is no separate REST API — the UI calls these functions directly.

Every action:
1. Creates a Supabase server client
2. Calls `supabase.auth.getUser()` — throws if not authenticated
3. Performs the DB operation scoped to `user.id`
4. Calls `revalidatePath()` so Next.js re-fetches the page data

### Task Actions

| Function | What it does |
|----------|-------------|
| `getTasks(filter)` | Fetch all/active/completed tasks for current user |
| `getTodayTasks()` | Fetch today's tasks |
| `addTask(title, priority)` | Insert a new task |
| `toggleTaskComplete(id, isCompleted)` | Flip is_completed |
| `deleteTask(id)` | Delete a task by ID |
| `getTasksForLast7Days()` | Returns 7 days of task data for the chart |
| `getDashboardData()` | Returns tasks + workout + weekly stats in one call |

### Workout Actions

| Function | What it does |
|----------|-------------|
| `getTodayWorkout()` | Get today's workout row (if exists) |
| `getTodayWorkoutWithDetails()` | Today's workout + exercises + sets (nested) |
| `getAllWorkoutsWithDetails()` | All past workouts + exercises + sets |
| `startWorkout()` | Upsert today's workout (creates if not exists) |
| `addExercise(name)` | Add exercise to today's workout (calls startWorkout first) |
| `addSet(exerciseId, weight, reps)` | Add a set to an exercise |
| `toggleSetComplete(id)` | Flip is_completed on a set |
| `getPersonalRecords()` | Best weight×reps combo per exercise name |

### Cardio Actions

| Function | What it does |
|----------|-------------|
| `getCardioLogs()` | All cardio logs for user |
| `addCardioLog(...)` | Insert new cardio log |
| `updateCardioLog(id, patch)` | Partial update of a cardio log |
| `deleteCardioLog(id)` | Delete a cardio log |

#### `/auth/signup` — Sign up
- Email/password registration
- On success, redirects to login or dashboard

- Log sessions with: activity type, duration, date, distance, steps, notes
- Conditional fields (distance/steps shown only for relevant activity types)
- Edit and delete existing logs
- Summary stats: total sessions, total time, total distance, total steps

## 10. Components

### `AppShell.tsx`

The main navigation wrapper for all authenticated pages. Renders:

- **Desktop:** fixed 60px sidebar on the left with icon-only nav links
- **Mobile:** hamburger menu → slide-in drawer from left
- **Mobile:** bottom tab bar with 4 navigation items
- Header with app name and a "live" green indicator dot
- Theme picker and sign-out button at the bottom of the sidebar

### `ThemePicker.tsx`

A dropdown button that lets users switch between 5 themes. Saves to `localStorage` under key `fittrack-theme` and applies `data-theme="..."` to the `<html>` element.

### `DashboardClient.tsx`

A client component (uses `'use client'`) that receives pre-fetched server data as props and renders:

- Animated count-up numbers using `requestAnimationFrame`
- Three stat cards: Today's Tasks, Gym Workout, Weekly Total
- `WeeklyProgressChart` bar chart
- Framer Motion animations on card hover/mount

### `WeeklyProgressChart.tsx`

A `Recharts` `BarChart` rendered inside a `ResponsiveContainer`. Shows 7 days (Sun → Sat) of completed task counts. Bars use SVG gradient fills — taller bars get a brighter gradient, zero bars show as dashed outlines.

### `WorkoutActions.tsx`

A single button that links to `/gym`. Shows "Start Workout" if no workout exists today, "View Workout" if one does. Colour changes accordingly.

## 11. Theme System

The app ships with 5 themes: **dark**, **light**, **acid**, **floors**, **liberty**.

### How it works

1. `ThemeContext.tsx` creates a React context with `theme` state and `setTheme()`
2. `src/app/layout.tsx` wraps the app in `<ThemeProvider>`
3. When theme changes, the context sets `document.documentElement.setAttribute('data-theme', theme)`
4. Tailwind CSS classes and CSS variables respond to the `data-theme` attribute
5. Theme is persisted to `localStorage` (key: `fittrack-theme`) and rehydrated on load

### Adding a new theme

Add the theme name to the `themes` array in `ThemeContext.tsx`, then add corresponding CSS variable overrides for `[data-theme="yourtheme"]` in `globals.css`.

## 13. Data Flow Examples

### Adding a Task

```
User types task title + selects priority
    ↓
Clicks "Add Task" in todo/page.tsx
    ↓
Calls addTask(title, priority)  ← Server Action in actions.ts
    ↓
Server: getUser() → get auth.uid()
    ↓
Server: INSERT INTO tasks (user_id, title, priority) VALUES (...)
    ↓
Server: revalidatePath('/todo') + revalidatePath('/dashboard')
    ↓
Next.js re-renders /todo with updated task list
    ↓
UI shows new task in list
```

### Logging a Workout Set

```
User fills "Exercise Name" input on /gym
    ↓
Calls addExercise(name)
    ↓
Server: startWorkout() → upserts today's workout row (creates if missing)
    ↓
Server: INSERT INTO exercises (workout_id, name)
    ↓
UI shows new exercise with empty sets table

User enters weight + reps for a set
    ↓
Calls addSet(exerciseId, weight, reps)
    ↓
Server: INSERT INTO sets (exercise_id, weight, reps)
    ↓
UI shows new row in the exercise's sets table
```

### OAuth Sign In

```
User clicks "Sign in with Google"
    ↓
supabase.auth.signInWithOAuth({ provider: 'google' })
    ↓
Browser redirects to Google OAuth consent screen
    ↓
Google redirects to /auth/callback?code=...
    ↓
route.ts: supabase.auth.exchangeCodeForSession(code)
    ↓
Supabase sets auth cookies
    ↓
Route redirects to /dashboard
    ↓
proxy.ts validates cookies on every subsequent request
```

## 14. Security Model

| Layer | Mechanism |
|-------|-----------|
| Route protection | `proxy.ts` middleware — unauthenticated requests redirected to `/` |
| Database access | Supabase RLS — every query is scoped to `auth.uid() = user_id` |
| Server Actions | Each action calls `getUser()` first — no user = no DB access |
| Secrets | `.env.local` is gitignored — never committed |
| Cookie security | Managed by Supabase SSR library — HTTPOnly, SameSite defaults |
| Public keys | Anon key + URL are safe to expose (RLS is the security layer) |

**The anon key is NOT a secret** — it only lets users do what RLS policies allow (i.e., their own data only). The actual security is RLS at the database level.
